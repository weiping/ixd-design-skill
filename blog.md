# 用 AI 做交互设计：ixd-design Skill 完全指南

> 一套让 AI 变成你的交互设计搭档的开源工具，8 个阶段、苏格拉底式发现、依赖感知断点恢复、评审门禁——从产品构思到可交互原型，全程 AI 驱动。

做过产品设计的人都知道，交互设计是一个既需要系统性思维、又极度依赖细节的工作。传统工具链通常是这样的：用 Figma 画原型、用墨刀做交互连线、用 Axure 写交互说明、用 Word 写设计文档——四个工具、四种格式，来回切换。

如果我告诉你，现在用一行命令，就能让 AI 帮你走完从需求分析到可点击原型的全部流程呢？

这就是 **ixd-design**——一个兼容 [AgentSkills](https://agentskills.io) 标准的交互设计 Skill。它不是一个简单的提示词集合，而是一套完整的设计方法论引擎，内置了 8 个阶段的设计流程、苏格拉底式发现机制、依赖感知的断点恢复系统、以及带三振出局的评审门禁。支持 Pi、OpenClaw、Claude Code、以及任何支持 Markdown 指令的 LLM。

本文将详细拆解这套系统的设计理念和使用方法，并用一个完整的「毛球社区」宠物 App 示例带你走一遍全流程。

## 什么是 ixd-design Skill？

ixd-design 是一个 AgentSkills 标准的交互设计技能插件，它把专业交互设计师的工作流程编码成了一套结构化指令。安装后，AI 就像拥有了一位资深设计搭档——你只需要描述产品构想，它就能按照专业流程一步步产出全套设计文档。

它能产出的成果物，对标的是墨刀和 Figma 的核心功能：

| 你在墨刀/Figma 里做的 | ixd-design 对应的产出 |
|----------------------|---------------------|
| 页面树 | Mermaid 信息架构图 + 穷举页面清单 |
| 流程连线 | Mermaid 流程图 + 步骤表 |
| 交互标注 | 逐页 10 节交互说明（含交互走查 + 微交互） |
| 组件库 | 组件规范 + Design Token 体系 |
| 设计稿/视觉标注 | 10 维度视觉方案 + Style Tile |
| 原型演示 | 可交互 HTML 原型（跨平台产品输出移动端 + 桌面端双原型） |
| 设计文档导出 | 完整设计说明书（含评审报告） |

**核心特性一览**

**8 阶段全流程**：从产品上下文到交付文档，覆盖交互+视觉设计全链路。

**苏格拉底式发现**：阶段一开始前，AI 分析你已提供的信息，针对 8 个关键维度自适应生成 0-8 个精准问题——信息充分时零问题直接开工。

**评审门禁**：阶段八交付后，强制 47 项走查 + 6 角色评审，未通过进入修复循环，三轮不过硬阻塞。

**依赖感知断点恢复**：每个阶段只加载它直接依赖的前序产出，不浪费 token。支持自然语言和自动检测两种恢复方式。

**22 类页面穷举**：覆盖移动端 18 类 + 桌面端 4 类专属页面类型，搭配生命周期校验确保不遗漏。

**PC 桌面端全链路支持**：从信息架构到原型，每个阶段都有 PC 客户端专属设计内容。

### 阶段一：产品上下文（含苏格拉底式发现）

这是整个设计的地基，也是 ixd-design 最智能的阶段。

#### Step A：苏格拉底式发现

传统做法是给设计师一张固定清单让他逐项填写。ixd-design 不这么做——它会先分析你已经说了什么，然后只问你没说的。

系统内部定义了 8 个信息维度，按对后续阶段的影响程度排序：

| # | 维度 | 为什么重要 | 影响的阶段 |
|---|------|-----------|-----------|
| D1 | 产品平台 | 决定页面类型、导航、组件、原型策略 | P2-P8 全部 |
| D2 | 核心功能模块 | 驱动页面清单和用户流程 | P2, P3, P4 |
| D3 | 目标用户与场景 | 驱动流程设计和交互复杂度 | P3, P4 |
| D4 | 设计风格方向 | 驱动视觉系统和组件风格 | P5, P6, P7 |
| D5 | 技术框架 | 约束组件方案和原型实现 | P5, P7 |
| D6 | 设计规范基底 | 决定组件库基础 | P5, P6 |
| D7 | 竞品/参考产品 | 校准设计方向和范围 | P1, P2, P6 |
| D8 | 产品定位与差异化 | 指导设计原则和取舍 | P1, 全部 |

对于每个维度，AI 的判断逻辑是：
- **已明确提供** → 跳过，不问
- **模糊/部分** → 生成简短追问（不是完整模板）
- **完全缺失** → 使用完整问题模板

如果两个维度的信息可以合并问，就合并。最终问题数量在 0-8 之间——如果你给了一份详细的 PRD，可能一个问题都不需要问。

每个选择题都带 ⭐ 推荐选项。如果你说"跳过"或"你决定"，AI 会使用推荐选项并在产出物中标注。

举个例子。你说："帮我设计一个类似 Notion 的协作笔记工具，React 技术栈"。AI 的分析：

```
D1 平台: 缺失 → 问（Notion 有桌面端，但你可能只做 Web）
D2 功能: 部分（"协作笔记"，但具体模块不明）→ 简短追问
D3 用户: 缺失 → 问
D4 风格: 可从"类似 Notion"推断 → 跳过
D5 技术: 已解决（React）→ 跳过
D6 设计规范: 缺失 → 问
D7 竞品: 已解决（Notion）→ 跳过
D8 定位: 部分（"类似 Notion"但差异不明）→ 简短追问

结果：4 个问题（不是 8 个）
```

一个一个问，每个问题都带进度指示器 `📋 问题 (2/4)`。

#### Step B：产出物生成

收集完信息后，AI 产出四样东西：
- 200 字以内的产品设计摘要
- 3-5 个设计挑战
- 3-5 条设计原则（具体可操作的，如"核心功能三步可达"，不是空泛的"用户友好"）
- 初步的视觉方向关键词

同时检测平台类型。如果包含 PC 客户端，会在上下文文档中标记——后续所有阶段自动启用 PC 专属内容。如果是跨平台（移动端 + 桌面端），设置 `crossPlatform: true`，阶段七会输出双原型。

### 阶段二：信息架构

这个阶段回答一个关键问题：**产品一共有多少个页面？它们之间是什么关系？**

ixd-design 内置了一套 **22 类页面类型分类法**，确保你不会遗漏任何类型的页面：

**核心内容类**（5 种）：聚合页(Hub)、列表页(List)、详情页(Detail)、搜索页(Search)、筛选/排序页(Filter)

**表单与输入类**（3 种）：表单页(Form)、多步表单(Wizard)、选择器页(Picker)

**反馈与结果类**（2 种）：结果页(Result)、空态页(Empty)

**账户与系统类**（4 种）：登录/注册页(Auth)、个人中心(Profile)、设置页(Settings)、关于/协议页(About)

**引导与过渡类**（3 种）：启动页/闪屏(Splash)、引导页(Onboarding)、过渡/加载页(Transition)

**覆盖层类**（1 种）：弹窗/浮层(Overlay)

**桌面端专属类**（4 种）：主窗口/工作区(Workspace)、侧边面板(Side Panel)、偏好设置窗口(Preferences)、托盘/菜单栏(Tray/Menu Bar)

除了分类法，还有一套 **生命周期校验清单** 帮你查漏：

1. 用户生命周期：安装 → 启动 → 注册 → 登录 → 使用 → 设置 → 退出 → 卸载
2. 内容生命周期：创建 → 编辑 → 发布 → 审核 → 展示 → 归档 → 删除
3. 交易生命周期：浏览 → 加购 → 下单 → 支付 → 配送 → 签收 → 评价 → 退款
4. 账户管理：注册/登录/找回密码/绑定/实名/注销
5. 消息通知：系统/业务/聊天/推送落地页
6. 异常兜底：404/网络错误/服务维护/版本过低/封禁
7. 法规合规：用户协议/隐私政策/权限申请说明
8. **桌面端生命周期**（如适用）：安装/卸载→首次启动→窗口管理→托盘行为→自动更新→快捷键→系统集成（文件关联/右键菜单/协议唤起）

导航结构也按平台区分：移动端用底部 Tab + 顶部导航，桌面端用侧边栏 + 菜单栏 + 工具栏。跨平台产品会输出各平台导航映射关系。

产出物是一张完整的页面清单表、一张 Mermaid 结构图、导航设计和全局组件清单。

### 阶段三：用户流程

有了页面地图，接下来要画出用户怎么在这些页面之间走。

AI 会为 3-5 个核心任务各画一张 Mermaid 流程图，包含正常路径和异常路径。同时产出步骤表，精确到每一步的用户操作、系统响应、异常处理和耗时预估。

如果平台包含 PC 客户端，还会增加桌面端专属流程：冷启动流程（下载→安装向导→首次启动→配置向导→主窗口）和自动更新流程。

这个阶段还有一个隐藏价值：画流程图的过程往往会发现阶段二遗漏的页面。比如画"退款流程"时才发现缺了"退款进度查询页"——系统会提醒你回去补充。

### 阶段四：逐页交互说明

这是整个体系的**核心产出**，直接对标墨刀的交互标注功能。

每个页面会产出完整的 **10 节**交互说明：

| 节 | 名称 | 内容 |
|----|------|------|
| 1 | 页面概述 | 核心功能、在产品中的位置 |
| 2 | 页面布局结构 | 用 ASCII 线框图描述分区（移动端单栏 / 桌面端多面板） |
| 3 | 组件清单 | 表格列出所有交互组件 |
| 4 | 交互行为详细说明 | 每个组件的点击、长按、滑动、**悬停、右键、键盘快捷键、拖拽**、输入、禁用态、反馈 |
| 5 | 页面状态流转 | 7 种状态：默认态 → 加载态 → 空态 → 错误态 → 无权限态 → 部分加载态 → 编辑态 |
| 6 | 动效说明 | 页面转场、元素动画、手势交互 |
| 7 | 数据加载策略 | 分页、缓存、刷新、离线 |
| 8 | 适配说明 | PC 窗口规范（最小尺寸、可缩放、面板布局、标题栏样式）、平板、小屏、横屏、深色模式、无障碍、多窗口 |
| 9 | **交互走查** | 强制自查：对照 47 项走查清单逐项检查，输出评分表。如有不通过项，先回去修改 1-8 节 |
| 10 | **微交互说明** | 识别页面中的微交互场景（点赞、收藏、删除、拖拽排序等），描述触发条件、视觉变化、动画参数、触觉/声音反馈、状态回退、CSS 伪代码 |

相比传统的 8 节交互说明，第 9 节和第 10 节是关键新增。第 9 节的交互走查让 AI 在产出每个页面时就进行质量自查，而不是等到最后才发现问题。第 10 节的微交互说明填补了传统交互文档中最容易被忽略的细节——那些让产品"有质感"的小动画。

由于页面数量可能很多（几十甚至上百个），阶段四采用**分批处理**策略——每次处理 3-5 个页面，保存为 `phase4-page-specs/batch-<N>.md`，确认后再做下一批。

### 阶段五：组件规范

从所有页面的交互说明中提炼出可复用的组件，形成项目级组件库。

包含两层内容：一层是具体的组件规范（每个组件的属性、变体、尺寸、状态、交互规则、使用场景）；另一层是 Design Token 体系（颜色、字体、间距、圆角、阴影、动效曲线——这些是整个设计系统的原子变量）。

如果产品面向 PC 客户端，组件规范还会包含桌面端专属组件：

- Window chrome（标题栏、窗口控制按钮）
- 工具栏按钮与工具栏
- 右键菜单 / 上下文菜单
- 可拖拽分割面板
- 数据表格（可排序、可调列宽、多选）
- 树形视图
- 命令面板（Cmd+K / Ctrl+K）
- 带快捷键徽章的 Tooltip
- 状态栏组件
- 系统托盘面板

### 阶段六：视觉设计

组件规范定义了"骨架"，视觉设计给骨架"穿衣服"。

覆盖 10 个维度：

1. **色彩系统**：品牌色、功能色（成功/警告/错误/信息）、中性色阶（8-10 级灰度），每个色值同时提供 Light 和 Dark 模式
2. **字体系统**：字体选择（中文/英文/数字）、8 级字号阶梯、排版规则。桌面端专属字体降级方案：Windows 雅黑链（Microsoft YaHei → SimHei → sans-serif）、macOS 苹方链（PingFang SC → Hiragino Sans GB → sans-serif）
3. **图标系统**：风格定义（线性/面性/双色）、尺寸规范、图标库推荐。桌面端专属尺寸：侧边栏 20px、工具栏 16-20px、系统托盘 16-22px、菜单项 16px
4. **插图系统**：风格、各种空态和引导场景的插图建议
5. **阴影层级**：5 级阴影，从平铺到浮层
6. **圆角系统**：统一的圆角值映射
7. **间距栅格**：基于 4px 网格的间距体系。桌面端栅格：lg(1024-1279px)、xl(1280-1439px)、2xl(≥1440px)，12-16 栏
8. **关键页面视觉标注**：3-5 个核心页面的详细视觉标注（跨平台产品分别标注移动端和桌面端）
9. **深色模式方案**：完整的颜色映射表。桌面端支持跟随系统主题自动切换 + Windows 高对比度模式
10. **动效规范**：缓动曲线、时长规范

还可以选择生成一个 **Style Tile**——一个 HTML 页面，把所有视觉 Token 可视化展示出来，支持 Light/Dark 切换，方便团队对齐视觉感受。

### 阶段七：可交互原型

这是 AI 相比传统工具的**差异化优势**所在——直接生成可点击的高保真 HTML 原型。

#### 技术栈和脚手架

原型基于 **React 18 + TypeScript + Tailwind CSS + shadcn/ui** 构建，最终打包成一个可在浏览器直接打开的单文件 HTML。

ixd-design 提供了一套开箱即用的脚手架脚本（`scripts/`），一行命令生成完整配置好的 React 项目：

```bash
./scripts/init-artifact.sh <项目名>
```

脚手架自动完成：Vite 项目创建、Tailwind 配置（含 CSS 变量、深色模式、`scrollbar-hide`、`safe-area` 等工具类）、shadcn/ui 全套组件安装（40+ 组件）、路径别名配置、**Vitest + Testing Library 测试框架**安装。项目初始化后就能直接写代码和测试。

#### 布局组件：Slot 模式（v2.8 重构）

原型的核心是三个布局组件，以**单一可信源**（`scripts/templates/`）管理：

**PhoneFrame**（移动端设备模拟）：提供 iPhone 14 物理框架，内置手机 Chrome（状态栏、灵动岛、Home Indicator）。Tab Bar 通过 `tabBar` slot 传入，支持任意自定义的 `AppTabBar` 组件：

```tsx
// ✅ Slot 模式 — Tab Bar 由调用者完全控制
<PhoneFrame theme={theme} tabBar={<AppTabBar currentPage={page} onNavigate={setPage} />}>
  <Home />
</PhoneFrame>

// ❌ 旧 API（已废弃）
<PhoneFrame tabs={tabs} activeTab="home" onTabChange={setPage}>
```

**WindowFrame**（桌面端设备模拟）：提供 macOS 风格窗口框架，内置标题栏。侧边栏通过 `sidebar` slot 传入：

```tsx
<WindowFrame sidebar={<AppSidebar currentPage={page} onNavigate={setPage} />} title="Dashboard">
  <Dashboard />
</WindowFrame>
```

这个 Slot 设计的好处：AppTabBar 和 AppSidebar 完全自定义（支持凸起按钮、徽标、折叠动画），PhoneFrame/WindowFrame 只负责物理框架，职责分离清晰。

**PrototypeShell**（原型外壳）：最外层包裹，提供项目名展示、页面导航器、浅/深色模式切换。它不包含任何设备框架——设备框架由各页面内部决定。

#### 职责划分（避免常见错误）

移动端：

| UI 层 | 由谁负责 |
|-------|---------|
| 状态栏 / 灵动岛 / Home Indicator | **PhoneFrame** 内置 |
| Tab Bar | **`tabBar` slot** → 自定义 `AppTabBar` |
| 顶部导航栏（标题/返回/搜索） | **页面 children** |
| 页面内容 | **页面 children** |

桌面端：

| UI 层 | 由谁负责 |
|-------|---------|
| 标题栏（红黄绿交通灯） | **WindowFrame** 内置 |
| 侧边栏导航 | **`sidebar` slot** → 自定义 `AppSidebar` |
| 菜单栏 + 工具栏（Word 风格） | **页面 children** — `flex flex-col h-full` 包裹 |
| 页面内容 | **页面 children** |

Word 风格（菜单栏 + 工具栏固定在上方，内容区可滚动）的实现方式是在 children 中用 `flex flex-col h-full` 包裹：

```tsx
<WindowFrame sidebar={<AppSidebar />}>
  <div className="flex flex-col h-full">
    <AppMenuBar />                     {/* flex-shrink-0 — 固定 */}
    <AppToolbar />                     {/* flex-shrink-0 — 固定 */}
    <div className="flex-1 overflow-y-auto p-6"> {/* 可滚动 */}
      {/* 页面内容 */}
    </div>
  </div>
</WindowFrame>
```

WindowFrame 不需要改动，菜单栏/工具栏完全在页面层控制。

#### TDD 优先开发流程（v2.6 新增，v2.8 强化）

阶段七不是"先写页面后测试"，而是 **TDD（测试驱动开发）**：

```
写测试 (RED → pnpm test 失败)
     ↓
实现页面 (GREEN → pnpm test 通过)
     ↓
下一个页面...
完成一批后：Bundle → 走查
```

PhoneFrame 和 WindowFrame 都内置了 `data-testid`（分别是 `"phone-frame"` 和 `"window-frame"`），让测试可以验证页面正确使用了设备框架：

```tsx
describe('Home', () => {
  it('renders without crashing', () => {
    expect(() => render(<Home />)).not.toThrow();
  });
  it('wraps in PhoneFrame', () => {
    const { getByTestId } = render(<Home />);
    expect(getByTestId('phone-frame')).toBeInTheDocument(); // data-testid 内置
  });
});
```

`pnpm test` 是 watch 模式，开发时持续运行；`pnpm test:run` 是单次运行，在打包前确认所有测试通过。

#### 分批生成 + 统一评审报告

原型按批次（每批 3-5 个页面）生成，每批完成后：

1. 运行 `pnpm test:run` 确认 TDD 绿灯
2. 运行 `bundle-artifact.sh` 打包为 HTML
3. 验证设备框架：grep 确认页面数量与框架数量匹配
4. 运行工具二（47 项启发式检查）走查批次页面
5. 将评审结果**追加**写入 `doc/ixd/phase7-review-master.md` 的 `## Batch N` 节

注意：**所有批次共用一个评审文件**（增量追加），不再为每批单独生成文件。完整性检查的最终报告也追加到同一文件的 `## Final Completeness Check` 节。

#### 单平台产品

输出一个 `phase7-prototype.html` 文件。如果是移动端，用 390×844px 的手机框架模拟，具备 Tab 导航切换、页面跳转动画、表单实时校验、骨架屏加载模拟、空态/错误态切换、深色模式切换等能力。

#### 跨平台产品（重点）

对于同时面向移动端和桌面端的产品，ixd-design 采用**双原型策略**——输出两个独立的 HTML 文件，共享同一套 Design Token CSS 变量：

| 文件 | 视口 | 导航 | 交互重点 |
|------|------|------|---------|
| `phase7-prototype-mobile.html` | 390×844px（居中手机框架） | 底部 Tab + 顶部导航栏 | 触摸手势、下拉刷新、滑动操作 |
| `phase7-prototype-desktop.html` | 1280×800px（桌面窗口） | 侧边栏 + 顶部工具栏/菜单栏 | 悬停态、右键菜单、键盘快捷键、多面板拖拽、窗口标题栏 |

为什么不用一个响应式文件模拟两端？因为移动端和桌面端的信息架构、导航层级、信息密度差异太大。底部 Tab 变成侧边栏不是简单的 CSS 断点能解决的——它涉及完全不同的交互范式。两个独立原型更真实，也更容易让团队和客户感受到真实的使用体验。

桌面端原型的特殊处理：
- 满宽布局（非手机框架）
- 侧边栏导航（240px 展开 / 56px 折叠）
- 顶部工具栏（可选：Word 风格固定菜单栏 + 工具栏）
- 两栏或三栏主体布局
- 右键弹出上下文菜单
- 按钮/菜单项显示键盘快捷键提示
- lg/xl/2xl 断点窗口缩放响应
- 模拟窗口标题栏（关闭/最小化/最大化）
- Tab 焦点导航和 Esc 关闭

### 阶段八：设计交付 + 评审门禁

把前七个阶段的所有产出打包成一份正式的设计说明书。结构涵盖 7 个章节（设计概述、信息架构、用户流程、页面交互说明、组件规范、视觉设计、全局交互规范）加附录（原型链接、视觉样例板、资源清单、开发交接注意事项、页面穷举清单、**快捷键映射总表**）。

跨平台产品的交付文档每个章节都会分别描述移动端和桌面端的设计，桌面端专属内容贯穿全文。

但文档生成只是一半。接下来是整个流程的最后一关——

#### 评审门禁（三振出局）

交付文档保存后，强制执行两轮评审：

**第一步：47 项交互走查**

从设计文档中选取 3-5 个代表性页面，对照 47 项走查清单（覆盖 10 个维度：基础交互、页面状态、导航与流程、表单与输入、数据加载、内容展示、视觉与品牌、触摸与点击、桌面端专属、跨平台一致性）逐项评分。

**第二步：6 角色多视角评审**

分别从新手用户、重度用户、无障碍用户、前端开发、QA 测试、品牌设计师 6 个角色审视完整设计，每个角色给出 2 个赞同点 + 2 个改进项，并按 P0/P1/P2 标注优先级。

**通过标准**：走查无优先修复项 AND 评审无 P0/P1 改进项。P2（建议类）不阻塞，记录在报告中留待后续迭代。

**未通过怎么办？**

```
生成文档 → 走查 + 评审
                ↓
          通过？──→ ✅ 完成
            ↓ No
      溯源修复 → 重新评审（Round 2）
            ↓ Still No
      再次修复 → 重新评审（Round 3）
            ↓ Still No
      ⛔ 硬阻塞 — 需人工介入
```

修复不是在交付文档上打补丁，而是**溯源到源阶段文件**修改。比如页面交互有问题，回去改 `phase4-page-specs/` 对应的页面文件，然后同步更新交付文档，再重跑评审。

三轮还过不了？系统硬阻塞，输出剩余问题清单和建议的修复阶段，等待人工决策。这个机制杜绝了"AI 无限自我修复"的死循环。

评审报告保存在 `doc/ixd/phase8-review-round-<N>.md`。


## 依赖感知的断点恢复

交互设计项目往往持续数天甚至数周，不可能一次做完。ixd-design 设计了一套智能的断点恢复系统。

### 不是"加载全部"，而是"按需加载"

传统做法是恢复时把所有历史产出全塞进上下文。ixd-design 不这么做——每个阶段有明确的依赖表：

| 要开始的阶段 | 必须完整加载 | 只看摘要 | 跳过 |
|-------------|------------|---------|------|
| P4 | P2, P3 | P1 | — |
| P5 | P4（当前批次） | P1, P2 | P3 |
| P6 | P1, P5 | P2 | P3, P4 |
| P7 | P5, P6, P4（当前原型页面） | P1, P2 | P3 |
| P8 | P1, P2, P3, P5, P6, P4（按章节） | — | — |

为什么 P5 不需要加载 P3？因为 P3 的用户流程信息已经内化在了 P4 的逐页交互说明中，加载 P4 就够了。

为什么 P7 不需要加载全部 P4？因为阶段四可能有几十个页面文件，原型每次只做一批页面，只加载当前批次的页面文件即可。

`progress.json` 中每个阶段都有结构化摘要（≤200 字），下游阶段通过摘要获得间接依赖的关键信息（比如产品名、平台、页面数量），而不需要完整加载原文。

### 恢复方式

**方式一：自然语言**

直接用中文或英文说就行：

```
"从阶段六开始，之前的交互说明和组件规范都做好了"
"接着上次的做，做到页面交互说明了"
"start from phase 4"
"continue where we left off"
"我有PRD了，直接开始信息架构"
```

AI 会根据你的描述判断从哪里开始。

**方式二：自动检测**

如果 `doc/ixd/progress.json` 存在，直接读取它作为唯一真相来源。如果没有 progress.json，AI 会从你提供的内容中推断完成状态——检测到页面清单和流程图就知道阶段二三已完成，建议从阶段四开始。当然，它会先确认判断是否正确。

两者通过单一可信源原则解耦：`templates/` 目录下的三个组件是权威版本，`init-artifact.sh` 在初始化时将其复制到项目中，`phase7-prototype.md` 的文档始终引用这些组件的 API——三处一致，修改模板只需改一处。

## 实战示例：设计一个「毛球社区」宠物 App

让我们用一个完整的示例来看看 ixd-design 的实际使用体验。假设我们要设计一个名为「毛球社区」的宠物生活 App，支持 iOS、Android 和小程序三端。

### 启动项目

在你的 AI 编程工具中输入：

```
帮我设计一个宠物社区App，名字叫"毛球社区"，
面向养猫养狗的年轻人，核心功能包括：宠物社交分享、
在线问诊、宠物用品商城、附近宠物服务。
需要同时支持 iOS、Android 和微信小程序。
```

AI 识别到设计请求，自动触发 ixd-design skill，进入阶段一。

### 苏格拉底式发现运行示例

AI 分析你的输入，对照 8 个维度：

```
D1 平台: 已解决（iOS/Android/小程序）→ 跳过
D2 功能: 已解决（社交、问诊、商城、附近服务）→ 跳过
D3 用户: 部分（"养猫养狗的年轻人"，但具体场景不明）→ 简短追问
D4 风格: 缺失 → 问
D5 技术: 缺失 → 问
D6 设计规范: 缺失 → 合并到 D5 一起问
D7 竞品: 缺失 → 问
D8 定位: 缺失 → 问

结果：4 个问题
```

> 📋 问题 (1/4)
> 你的用户除了"养猫养狗的年轻人"，使用场景是怎样的？（决定用户流程设计和交互复杂度）
> 例如："① 宠物主人 — 通勤时刷社区动态，周末找附近宠物店；② 宠物医生 — 在线接诊、开处方"

你回答："主要是宠物主人，在家刷动态、出门找服务。暂时不做医生端。"

> 📋 问题 (2/4)
> 你期望的设计风格方向？
>   A. 🌿 温暖人文 — 柔和配色，自然元素（如 Calm、小红书）⭐ 推荐：契合宠物社区的温暖调性
>   B. 🎨 活泼多彩 — 丰富色彩，圆润元素
>   C. 🧊 简约克制 — 大量留白，信息密度低
>   D. 其他

你选 A。

> 📋 问题 (3/4)
> 前端技术栈和设计规范基底？（影响组件规范和原型实现方式）
>   A. Flutter + 自建设计系统  ⭐ 推荐：三端统一体验
>   B. React Native + Ant Design Mobile
>   C. 原生各端分开 + 各平台 HIG
>   D. 尚未确定

你选 A。

> 📋 问题 (4/4)
> 有参考的竞品吗？和它们相比最大的不同是什么？
> 例如："参考小红书的社区氛围 + 丁香医生的问诊流程，但更强调本地化服务"

你回答后，AI 整理所有信息进入 Step B，产出设计上下文：

```text
## 产品设计摘要

「毛球社区」是一款面向 18-35 岁养宠年轻人的跨平台宠物生活 App，
核心解决"养宠信息碎片化、线上线下服务断裂"的问题。
基于 Flutter 构建三端统一体验，主要包含 4 大功能模块：
宠物社交分享、在线问诊、宠物商城、附近服务。

## 设计挑战

1. **内容与电商的平衡**：社区内容和商城都想要首页流量，
   如何避免"既不是好社区、也不是好商城"？
2. **三端一致性**：iOS/Android/小程序的导航范式不同，
   如何保持交互一致的同时尊重各平台习惯？
3. **从线上到线下**：附近服务模块需要连接线下商家，
   LBS 信息展示和预约流程的设计挑战。

## 设计原则

1. **三步可达**：核心功能从首页出发 3 步内必须可触达。
2. **内容优先**：首页以 UGC 内容为主角，商业内容不侵入信息流。
3. **温度感**：设计风格传递"温暖陪伴"的情感，避免冰冷的工具感。
```

注意：整个过程只问了 4 个问题，不到两分钟就进入了正式设计。如果你一开始就给了一份详细的 PRD，可能连一个问题都不需要问。

### 阶段八运行示例（评审门禁）

假设 AI 完成了交付文档生成。接下来自动进入评审门禁：

```text
# Phase 8 设计评审报告 — Round 1

**评审结果**: ❌ 未通过
```

## 最佳实践

经过反复使用，我总结了几条让 ixd-design 发挥最大价值的实践建议。

### 给足上下文，信息越多问题越少

苏格拉底式发现的核心逻辑是"只问不知道的"。你一开始提供的信息越多，AI 需要问的问题就越少。如果你有 PRD、竞品分析、甚至老板的原话，全部粘贴进去。AI 消化信息的能力远超人类，信息过载不是问题，信息不足才是。

### 阶段四分批进行，不要贪快

逐页交互说明是工作量最大的阶段。一个中型 App 可能有 40-60 个页面。一次让 AI 生成太多页面，质量会下降。

推荐策略：按模块分批，每批 3-5 个页面。第 9 节交互走查会在每个页面产出时自动进行质量检查，发现问题及时修正，再做下一批。

### 先视觉后原型

不要跳过阶段六直接做原型。没有视觉方案的原型只是灰色线框图，无法传递产品的品牌气质和情感调性。先定好色彩和字体，原型才会看起来像一个真实的产品。

### 善用辅助工具

在任何阶段说"帮我分析一下丁香医生的问诊交互"，AI 就会输出结构化的竞品分析。说"帮我走查一下首页"，AI 会逐项检查 47 项维度。说"用多角色视角评审一下支付流程"，AI 会从 6 个角色分别给出评价。

这些工具不限阶段，随时可用。

### 利用断点恢复应对长项目

一个完整产品的设计可能需要好几天。每次会话结束前，确认 `doc/ixd/progress.json` 已更新。下次继续时，AI 会自动读取 progress.json 和依赖文件，无缝恢复。

### 多端适配的处理策略

像「毛球社区」这样需要同时支持 iOS、Android 和小程序的产品，建议在阶段一就明确"多端差异处理原则"。在阶段四的每个页面交互说明中，第 8 节"适配说明"专门处理多端差异。阶段七的原型也会根据 `crossPlatform` 标志决定输出单原型还是双原型。

|------|-----------------|-----------------|

| 产出速度 | 一个页面通常需要几小时 | 一个页面几分钟（AI 生成+人工审查） |

| 质量保证 | 依赖人工评审 | TDD 测试 + 逐页走查（47 项）+ 交付评审门禁 |
| 原型能力 | 静态连线 | 真实 React + TypeScript 代码，可交互 |
| 原型测试 | 无 | Vitest + Testing Library，自动化冒烟测试 |

需要注意的是，ixd-design 的定位不是替代 Figma，而是**替代设计流程中最耗时的结构化工作**——信息架构、交互说明、组件规范、设计文档。像素级的视觉设计稿（图标绘制、插图创作、精确的布局微调）仍然需要设计师用专业工具完成。

## 写在最后

ixd-design 的本质是一个方法论引擎。它把"怎么做好交互设计"这个经验性的知识，编码成了可执行的结构化流程。8 个阶段是骨架，苏格拉底式发现是入口，评审门禁是出口，依赖感知的断点恢复让它真正可用于多日的实际项目。

在我的实际使用中，一个中等复杂度的 App（约 40 个页面），从零到交付完整设计文档和可交互原型，大约需要 4-6 个小时的人机协作时间。同样的工作量，纯人工通常需要 2-3 周。这不是因为 AI 跳过了什么环节，而是因为 AI 承担了大量的结构化产出工作，让人可以把精力集中在创意决策和质量把控上。

如果你是独立开发者、产品经理或者初创团队的设计师，ixd-design 可能是你把产品设计水准提升一个台阶的最高性价比工具。
